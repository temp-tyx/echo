# 01 — 背景与配置

[← 返回目录](./README.md)

## 场景

- **框架**：vLLM v1 + vLLM-Ascend
- **模型**：Qwen3.5-4B + MTP
- **加速**：ECHO（Enhanced Concurrency with High-confidence prUning）
- **调度**：`async_scheduling=true`（可选，async 需额外 draft sync）

ECHO 在 draft propose 阶段扩展步数（上限 = 配置的投机宽度），再按置信度 global top-k 剪枝，低置信 draft 置 `-1`，最后把剪枝后的真实 draft 数 sync 回 scheduler。

## 单旋钮配置（核心）

部署时**只需一个旋钮**：`speculative_config.num_speculative_tokens`。把它设成你期望的**最大投机宽度**即可。见 `examples/qwen3.5-attention.py`：

```python
speculative_config = {
    "method": "mtp",
    # 关键：开到你想要的最大投机宽度。
    # scheduler lookahead / KV 预留 / ACL graph 宽度 / decode_threshold
    # 以及各 draft buffer 都按这个值在进程启动时统一分配。
    # ECHO 在此上限内做 top-k 剪枝，不会超过它。
    "num_speculative_tokens": 7,
}
```

> 约束：`num_speculative_tokens` 必须 **≥ ECHO 实际会 verify 的最大宽度**。取相等最省。配小了 → buffer/kernel 尺寸不足 → eager 下 MTE DDR 越界、graph 下静默错算。

## 环境变量（`vllm_ascend/envs.py`）

| 变量 | 默认 | 含义 |
|------|------|------|
| `VLLM_ECHO_ENABLED` | `1` | 是否启用 ECHO |
| `VLLM_ECHO_K_MAX` | `5` | global top-k 保留的 draft 单元格上限 |
| `VLLM_ECHO_STEPS_MULTIPLIER` | `2` | propose 步数倍率，`draft_step ≈ min(2*k_max/bs, num_speculative_tokens)` |
| `VLLM_ECHO_DEBUG_LOSSLESS` | `0` | 打开后在 rejection 后 dump 无损性校验（见 04） |

> 已删除：`VLLM_ECHO_MAX_SPEC_NUM`。它过去用来单独表达 draft 上限,现已与 `num_speculative_tokens` 合并为一个旋钮。

## 三套宽度——必须一致（必读）

系统里存在三套「宽度」。**早期 bug 全部来自让它们不一致**：

| 名称 | 来源 | 最终要求 |
|------|------|---------|
| **config 宽度** | `num_speculative_tokens` | = 最大投机宽度（如 7）|
| **draft 宽度** | drafter propose 列数 | ≤ config，prune 后更少 |
| **runtime decode 宽度** | `uniform_decode_query_len`、`decode_threshold`、ACL graph、各 buffer | 由 config 派生（`1 + num_speculative_tokens`）|

**正确做法（最终）**：让 config 宽度 = 最大投机宽度，则后两者由 vLLM 原生逻辑**自动**按同一上限派生，三套天然一致，无需任何 runtime patch。

**错误做法（已废弃）**：config 保持 3，再在 runtime 逐个 patch decode 宽度去对齐 verify 的 6 token。永远补不全（见 02）。

## 一步 decode 数据流

```text
schedule(N)              按 num_lookahead = num_speculative_tokens 预留
  → execute_model(N)     target verify（decode 路径，宽度足够）
  → sample_tokens(N)     rejection sampling
  → propose(N)           draft + ECHO top-k prune（≤ config）
  → take_draft / sync    写回 engine.spec_token_ids（过滤 -1）
  → schedule(N+1)        下一步按 sync 后的真实 draft 数 schedule
```

要点：

- **propose 在 execute 之后**，本步 prune 结果给 **N+1** 用。
- **engine 只认自己的 `request.spec_token_ids`**；不能靠 worker 本地改 `scheduler_output`（旧 trim 方案，已删）。
- truth 必须写在 `request.spec_token_ids`（**sync**）。
