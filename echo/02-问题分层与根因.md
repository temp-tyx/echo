# 02 — 问题分层与根因

[← 返回目录](./README.md)

## 问题分层总览

```text
Layer 1  KV / slot_mapping     init 改 vllm_config=7 → 物理块分配与基线不一致
Layer 2  schedule 账本        engine spec=3, worker 硬扩到 6 → 重复 token
Layer 3  attention 路径       decode_threshold=4, max_query_len=6 → 误判 prefill → 质量差
```

三层独立、症状不同，需分层修，不能一锅端改 init config。

---

## Layer 1：KV 脏页 / slot_mapping 漂移

### 现象

- multi-concurrency 乱码、NaN
- 同 prompt、同 batch，ECHO 与基线 **slot_mapping 不同**（同 logical block，不同 physical block）
- 例：position 2673 → 基线 slot 19057，ECHO slot 31345

### 根因

`AscendSpecDecodeBaseProposer.__init__` 若执行：

```python
vllm_config.speculative_config.num_speculative_tokens = VLLM_ECHO_MAX_SPEC_NUM
```

会在 `NPUInputBatch` / scheduler / KV 初始化 **之前** 把全局 config 改成 7，整条 KV 分配路径与 MTP3 基线不一致。

### 修复

- init **不改** `vllm_config`
- 仅 drafter 内部 `self.num_speculative_tokens = max(7, config)`

---

## Layer 2：重复输出

### 现象

- 单并发输出重复 token
- `num_computed` 某步异常回退（如 2685→2682）
- reject log：`num_accepted=0, generated_len=1`

### log 铁证（不改 config）

```text
trim: echo_cu=5  old_sched=4  spec_before=3  new_sched=6  spec_after=3
reject: engine_spec=3  num_accepted=0  num_rejected=3
```

### log 对照（改 config=7，单并发正常）

```text
trim: echo_cu=5  old_sched=8  spec_before=7  new_sched=6  spec_after=5
reject: engine_spec=5  num_accepted=5  generated_len=6
```

### 根因

1. **`draft_token_ids_cpu` 只有 3 列**（按 MTP3 init）→ sync 到 engine 的 `spec_token_ids` 长度 ≤ 3
2. **execute_model trim** 按 `echo_cu=5` 把 `num_scheduled` 扩到 6，但 `scheduled_spec` 只有 3 个
3. **engine `update_from_output`** 仍按 **未 trim** 的原始 schedule（spec=3）算 rejection → `num_computed` 账本错乱

trim 在 worker 本地改 `scheduler_output`，**engine 看不见**，不能作为长期方案。

### 修复

| 改动 | 作用 |
|------|------|
| 扩 `draft_token_ids_cpu` 到 7 列 | sync 不被截断 |
| `take_draft_token_ids` 过滤 `-1` | engine 只收到有效 draft |
| **去掉** execute_model trim | 不再 worker/engine 双账本 |
| `patch_echo_async.py` | async 下 schedule 前 sync draft |
| `_compact_echo_draft_rows` | prune 后左对齐，async scatter 前缀正确 |

## Layer 3：输出质量差（语句不通顺）

### 现象

- 重复输出消失后，JSON/文本语义混乱，如：`"饮料瓶代码与人物微光——"`
- 不像乱码/NaN，更像 **错误 draft 被 accept**

### log 铁证（sync 已对齐后）

```text
target_forward max_query_len=6
runner_decode_threshold=4
attn_decode_threshold=4
uniform_decode_query_len=4
per_req_scheduled={'0-xxx': 6}  spec_lens={'0-xxx': 5}
```

### 根因

Attention 中 decode/prefill 划分（`split_decodes_and_prefills`）：

```python
if max_query_len <= decode_threshold:
    return num_reqs, 0, num_tokens, 0   # 全部当 decode
# 否则 query_lens > decode_threshold 的 req 当 prefill
```

`6 > 4` → **6-token spec verify 被当成 prefill** → kernel/metadata 错误 → rejection 接受错误 token。

与 topk 挖洞 **无关**（bs=1 且 cum log-prob 单调时，topk 必为前缀 step 0..4）。

与 init 改 config=7 的对照：

| | config=3 + sync | config=7 |
|--|-----------------|----------|
| max_query_len | 6 | 6 |
| decode_threshold | **4** | **8** |
| 判定 | prefill ✗ | decode ✓ |

### 修复

**不动 init config**，ECHO 开启时将 runtime 宽度提到 `1 + VLLM_ECHO_MAX_SPEC_NUM`（默认 8）：

- `uniform_decode_query_len`
- `decode_threshold`（runner + attention builders）
- `decode_token_per_req`
- `cudagraph_dispatcher.uniform_decode_query_len`
- `compiler_interface` static kernel 捕获范围

---

## 为何「init 改 config=7」单并发能好

不是 pruning 更准，而是 **三条宽度在 init 时都被设成 7/8**：

- CPU buffer / sync / schedule 7 宽
- trim 后 `new_sched=6, spec_after=5` 自洽
- `decode_threshold=8`，`6 ≤ 8` 走 decode

代价：scheduler `num_lookahead=7`，multi-concurrency KV 路径与 MTP3 基线分叉。

---

## 已排除的方向

| 方向 | 原因 |
|------|------|
| topk 中间挖洞（bs=1） | cum log-prob 单调时必为前缀 |
| worker forward input_ids 错位（后期） | 用户 log 确认对齐；问题是 engine/runtime 宽度 |
| execute trim 长期保留 | 只修 worker，不修 engine 账本 |
