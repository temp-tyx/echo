## 当前状态（2026-06-17）

多并发 mixed batch（decode + prefill / 多 decode）已定位并修复以下问题：

- prefill 被误 trim（Bug #6）
- `total_num_scheduled_tokens` 与 per-req schedule 不一致
- scheduler `num_computed` 乐观过计数（Bug #7）
- `echo_keep=0` 时 `input_ids=[0]`（Bug #8）
- 错误的 `update_num_computed_tokens` 放宽导致负 position（Bug #9，已回滚）

**请在远端同步最新 `model_runner_v1.py` 后复测。**

## 仍待观察 / 未完全验证

1. **长 prefill + 多图 multimodal** 与 ECHO trim 的交互（Qwen3.5 M-RoPE position 三维日志不易读，需对照 `num_computed` 而非单维 position 数值）
2. **async scheduling 下 `accepted_tokens={}`** 为预期行为（token 走 GPU `prev_sampled_token_ids`，不在 CPU list）
3. **req0 先 finish、req1 继续** 时 batch slot / `num_computed` 是否始终与 `req_id` 对齐（过计数修正应缓解 2681 漂移）
4. **完全剪枝（echo_keep=0）** 后 draft propose 返回 `-inf`，下一拍仅 1 token decode — 依赖 sample token 回退逻辑

## 若仍失败时的排查顺序

1. 确认 `[trim] after` 中 prefill req 的 `old_sched` 未被改为 `1`
2. 确认 `[trim] after` 的 `total` = 各 req `new_sched` 之和
3. 确认 solo req 的 `num_computed` 不是 `prompt_len + old_sched`（过计数未修正）
4. 确认 `[target_forward] positions` 无负值
5. 确认 `[target_forward] input_ids` 无 trailing `0`
