# 03 — 当前方案与代码改动

[← 返回目录](./README.md)

## 设计原则（最终）

1. **单旋钮**：部署只设 `speculative_config.num_speculative_tokens = 最大投机宽度`（如 7）。
2. **从启动起一致**：scheduler lookahead、KV 预留、ACL graph 宽度、`decode_threshold`、各 draft buffer 全部由这一个值在进程启动时统一派生 → 三套宽度天然一致。
3. **ECHO 在上限内剪枝**：drafter propose ≤ config，global top-k prune 后更少。
4. **sync 真实 draft 数**回 scheduler（async 经 `patch_echo_async`），保证 scheduled 宽度 == verify 宽度 == 已写 KV 宽度。
5. **不在 runtime patch 派生宽度**，不在 worker 改 `scheduler_output`。

```text
        ┌──────────────────────────────────────────────┐
        │  num_speculative_tokens = 7  (启动配置, 单旋钮) │
        └──────────────────────────────────────────────┘
              │ 进程启动时统一派生
   ┌──────────┼───────────┬───────────────┬──────────────┐
   ▼          ▼           ▼               ▼              ▼
scheduler   KV 预留   decode_threshold  ACL graph    draft buffer
lookahead=7           = 1+7 = 8         宽度=8        7 列
   └──────────┴───────────┴───────────────┴──────────────┘
                          ▼
        ECHO propose(≤7) → top-k prune → sync 真实数 → schedule(N+1)
```

## 为什么不能回退到 `22ee2248`

`22ee2248` 看似"单并发已经好了",但它是**靠运气好**,且把多并发的坑留着没填。回退会同时丢掉修复、捡回 bug：

1. **它的 config 改法是错的（worker-only 偷改）。**
   `22ee2248` 在 `eagle_proposer.__init__` 里执行
   `vllm_config.speculative_config.num_speculative_tokens = VLLM_ECHO_MAX_SPEC_NUM`。
   这只在 **worker 进程**、且在 **drafter 创建那一刻**才生效：
   - **时序坑**：很多派生宽度（KV 预留、`decode_token_per_req` 等）在此之前已算好，改了也来不及。
   - **进程不一致**：**engine 进程**的 scheduler 看不到 worker 改的值，lookahead 仍按启动配置（3）。
   单并发能跑只是因为 batch 简单、恰好没踩到不一致的分支。

2. **多并发 KV 脏页在 `22ee2248` 从未修复。**
   `22ee2248` 既没有 `patch_echo_async` 的 draft sync,也没有 `_copy_draft_token_ids_to_cpu` 的真实列数 + `-1` 补齐。
   结果:scheduled 宽度 ≠ 已写 KV 宽度 → attention 读到未写脏 slot（NaN）。**回退 = 直接把多并发 NaN 崩溃捡回来。**

3. **当前版本是 `22ee2248` 的严格超集。**
   - 单并发质量:与 `22ee2248` 持平（同样达成）。
   - 多并发:`22ee2248` ✗ → 当前 ✅。
   - 配置:`22ee2248` 需要"启动配 3 + 环境变量 7 + worker 偷改"三者拼凑;当前只需 `num_speculative_tokens=7` 一个旋钮。
   - eager:`22ee2248` 下宽度不一致仍可能 MTE 越界;当前按 max 分配，eager/graph 均正确。

   回退唯一"收益"是少几行 diff,代价是**重新引入多并发 NaN + 偷改 config 的两个隐患**,得不偿失。

## propose 路径注意点

```python
draft_step = min(max(2 * k_max / batch_size, 1), self.num_speculative_tokens)
self.drafter.num_speculative_tokens = draft_step   # 仅 propose 期间
# 不写 vllm_config / speculative_config（这是 22ee2248 的错误做法）
# propose 后恢复 drafter 宽度；scatter stride 用 draft_tensor.shape[1]
```

## async vs sync draft sync

| 模式 | sync 时机 |
|------|-----------|
| sync | vLLM 原生 `EngineCore.post_step` → `take_draft_token_ids`（已过滤 `-1`）|
| async | `patch_echo_async`：schedule **前** + post_step 各 sync 一次 |

async 下若不 sync，engine 仍按 placeholder 宽度 schedule，账本漂移。

## 性能说明

| 项 | 开销 |
|----|------|
| config=7 vs 3 | KV 预留、graph 宽度略增（按最大投机宽度，本就是要的能力）|
| sync + 过滤 `-1` | 原链路已有，略增 list 长度 |
| 去 trim / 去 runtime patch | 略减 CPU 操作 |

相对 `22ee2248`:多并发可用、配置更简，单并发算力路径基本一致。
