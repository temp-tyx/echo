## 验证通过标准

### 1. 配置一致（前提）

启动配置 `num_speculative_tokens = 最大投机宽度`（如 7）。由此原生派生：

```text
scheduler num_lookahead = 7
decode_threshold        = 1 + 7 = 8
uniform_decode_query_len= 8
draft buffer 列数        = 7
```

`max_query_len`（如 6）`<= decode_threshold`（8）→ spec verify 走 **decode** 路径，不会被误判 prefill。

### 2. schedule 对齐（重复输出）

engine schedule 与 ECHO prune 一致（靠 sync，**不靠 worker trim**）：

```text
per_req_scheduled ≈ 1 + 有效 draft 数   # 如 6
spec_lens ≈ 有效 draft 数               # 如 5
```

不应再出现 `new_sched=6, spec_after=3` 这种 worker/engine 双账本不一致。

### 3. rejection 健康

```text
engine_spec=5  num_accepted=2~5  generated_len=3~6
num_computed 单调推进，不应无故大幅回退
```

不应再出现 `engine_spec=3, num_accepted=0, generated_len=1`。

### 4. multi-concurrency

- 与基线比，同 logical block 的 **slot_mapping** 应一致；无 KV 脏页 / NaN。
- 无 `tensor(7) vs (5)` copy 报错（`_copy_draft_token_ids_to_cpu` 已按真实列数 + `-1` 补齐）。
- eager（`--enforce-eager`）下无 MTE DDR 越界。

## 无损性自检：`VLLM_ECHO_DEBUG_LOSSLESS`

设 `VLLM_ECHO_DEBUG_LOSSLESS=1`，在 rejection 后 dump（`_echo_dump_lossless`），用于区分两类问题：

- **(a) target logits 在某 draft index 之后变脏** → forward/graph/attn 宽度 bug
- **(b) acceptance 偏离 greedy 规则** → rejection sampler bug

日志格式：

```text
[ECHO lossless] num_draft=[5]
  draft        =[...]      # ECHO 剪枝后送 verify 的 draft token
  tgt@draft    =[...]      # 各 draft 位置上 target 的 argmax
  bonus        =[...]      # bonus 位置 argmax
  accepted     =[[...]]    # rejection 实际 commit 的 token
```

判定（greedy）：逐位置 `accepted[i]` 应等于 `tgt@draft[i]`，首个不符即停。若 commit 与 target argmax 不符会打：

```text
[ECHO lossless] VIOLATION req=R pos=P committed=X tgt_argmax=Y draft=Z
```

- 出现 VIOLATION 且 `committed != tgt_argmax` → 无损性被破坏，多半是宽度/kernel（看是否伴随 `max_query_len > decode_threshold`）。
- 无 VIOLATION 但输出仍差 → 往 draft 质量 / 模型本身查。
