## 验证通过标准

### 1. schedule 对齐（重复输出）

engine schedule 与 ECHO prune 一致，**无需 execute trim**：

```text
per_req_scheduled ≈ 1 + echo_cu     # 如 6
spec_lens ≈ echo_cu                 # 如 5
```

不应再出现：`new_sched=6, spec_after=3`。

### 2. decode 路径（输出质量）

target forward 前（`[ECHO diag] target_forward`）：

```text
max_query_len=6
runner_decode_threshold=8      # 或 ≥ max_query_len
attn_decode_threshold=8
uniform_decode_query_len=8
```

判定：`max_query_len <= decode_threshold` → 走 **decode**，不是 prefill。

### 3. rejection 健康

```text
engine_spec=5  num_accepted=2~5  generated_len=3~6
num_computed 不应无故大幅回退
```

不应再出现：`engine_spec=3, num_accepted=0, generated_len=1`（在 sync 正确后）。

### 4. multi-concurrency

- 同 prompt/batch 与基线比 **slot_mapping** 同 logical block 应一致（init 不改 config 的目标）
- 无 KV 脏页 / NaN

## 诊断 log 字段

### target_forward（`model_runner_v1.py`）

```text
[ECHO diag] target_forward max_query_len=6 runner_decode_threshold=8 attn_decode_threshold=8
uniform_decode_query_len=8 cudagraph_mode=NONE per_req_scheduled={...} spec_lens={...}
```

| 字段 | 含义 |
|------|------|
| `max_query_len` | 本步 attention max query len |
| `runner_decode_threshold` | runner 侧 decode/prefill 阈值 |
| `attn_decode_threshold` | attention builder 阈值 |
| `uniform_decode_query_len` | ACL uniform graph 宽度 |
| `per_req_scheduled` | 每 req `num_scheduled_tokens` |
| `spec_lens` | 每 req draft 数 |

prefill 步：`max_query_len` 很大（prompt 长），`spec_lens={}`，正常。
