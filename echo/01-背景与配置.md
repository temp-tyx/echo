# 01 — 背景与配置

[← 返回目录](./README.md)

## 场景

- **框架**：vLLM v1 + vLLM-Ascend
- **模型**：Qwen3.5-4B + MTP（target `num_speculative_tokens=3`）
- **加速**：ECHO（Enhanced Concurrency with High-confidence prOuning）
- **调度**：`async_scheduling=true`（可选，async 需额外 draft sync）

ECHO 在 draft propose 阶段扩展步数（最多 `VLLM_ECHO_MAX_SPEC_NUM`），再按置信度 global top-k 剪枝，低置信 draft 置 `-1`。

## 环境变量（`vllm_ascend/envs.py`）

| 变量 | 默认 | 含义 |
|------|------|------|
| `VLLM_ECHO_ENABLED` | `1` | 是否启用 ECHO |
| `VLLM_ECHO_K_MAX` | `5` | global top-k 保留的 draft 单元格上限 |
| `VLLM_ECHO_STEPS_MULTIPLIER` | `2` | propose 步数倍率，`draft_step = min(2*k_max/bs, draft_max)` |
| `VLLM_ECHO_MAX_SPEC_NUM` | `7` | draft proposer buffer 上限（**不改 target MTP 配置**） |

bs=1 时典型值：`draft_cols=7`，prune 后 `echo_cu=5`，verify 时 `num_scheduled=6`（1 sample + 5 draft）。

## 宽度概念（必读）

系统里存在 **三套「宽度」**，混用是大部分 bug 的来源：

| 名称 | init 时来源（MTP3） | ECHO 实际需要 |
|------|---------------------|---------------|
| **config 宽度** | `vllm_config.speculative_config.num_speculative_tokens = 3` | 保持 3（scheduler `num_lookahead`、KV 预留） |
| **draft 宽度** | drafter `num_speculative_tokens = max(7, 3)` | propose 7 列，prune 后有效 5 |
| **runtime decode 宽度** | `uniform_decode_query_len = 4`，`decode_threshold = 4` | verify 6 token 时需要 **≥ 6**（稳妥用 8） |

**错误做法**：init 时 `vllm_config.speculative_config.num_speculative_tokens = 7`  
→ 单并发质量正常，但 multi-concurrency KV / slot_mapping 物理块漂移。

**正确做法**：config 保持 3；draft/sync/runtime 三条链分别对齐。

## 一步 decode 数据流

```text
schedule(N)
  → execute_model(N)     target verify
  → sample_tokens(N)       rejection sampling
  → propose(N)             draft + ECHO prune
  → take_draft / sync      写回 engine.spec_token_ids
  → schedule(N+1)          下一步按 sync 后的 draft 数 schedule
```

要点：

- **propose 在 execute 之后**，本步 prune 结果给 **N+1** 用。
- **engine 只看自己的 `scheduler_output`**，worker 本地改 `scheduler_output` 不会更新 rejection 账本。
- **truth 应写在 `request.spec_token_ids`**（sync），而不是 execute 前 trim。
