## 3. `vllm_ascend/worker/model_runner_v1.py`

### 3.1 实例变量

```python
self.echo_cu_draft_tokens: dict[str, int] = {}      # req_id -> 保留 draft 数
self.echo_draft_tokens: dict[str, list[int]] = {}   # req_id -> 保留 draft token
self.echo_prev_scheduled: dict[str, int] = {}       # trim 前 scheduler 调度量（old_sched）
self.echo_overcount_pending: dict[str, int] = {}    # 待修正的 num_computed 过计数
```

### 3.2 `_update_states`（override）— trim 入口

trim 从 `execute_model` 入口移至 `_update_states` **开头**，在 `super()._update_states()` 之前执行：

```python
def _update_states(self, scheduler_output):
    if envs.VLLM_ECHO_ENABLED and self.echo_cu_draft_tokens:
        self._apply_echo_scheduler_trim(scheduler_output)

    if envs.VLLM_ECHO_ENABLED and self.echo_overcount_pending:
        # 修正 scheduler 乐观 num_computed 过计数
        for req_id, over in self.echo_overcount_pending.items():
            req_data.num_computed_tokens[i] -= over
            self.requests[req_id].num_computed_tokens -= over

    return super()._update_states(scheduler_output)
```

**动机**：

1. 保证 `update_req_spec_token_ids` 读到的是 **trim 后** 的 spec，使 `prev_num_draft_len` 与 worker 实际执行一致
2. 在状态更新前修正 scheduler 传来的 `num_computed_tokens` 过计数
3. 避免在 `execute_model` 入口重复 trim

### 3.3 `_apply_echo_scheduler_trim`

对每个 `req_id`：

1. **跳过未参与上一轮 propose 的请求**（prefill / chunked-prefill 不在 `echo_cu_draft_tokens` 中）
2. 从 `echo_cu_draft_tokens` / `echo_draft_tokens` 取 keep 信息
3. `kept_spec_tokens = proposed_drafts[:draft_token_num]`
4. 更新 `scheduled_spec_decode_tokens`、`num_scheduled_tokens`
5. 记录 `echo_prev_scheduled[req_id] = old_num_scheduled_tokens`（供过计数修正）
6. `total_num_scheduled_tokens = sum(num_scheduled_tokens.values())`

```python
if req_id not in self.echo_cu_draft_tokens:
    continue  # 不 trim prefill 请求

draft_token_num = self.echo_cu_draft_tokens[req_id]
proposed_drafts = self.echo_draft_tokens.get(req_id, [])
kept_spec_tokens = proposed_drafts[:draft_token_num]
new_num_scheduled_tokens = len(kept_spec_tokens) + 1
self.echo_prev_scheduled[req_id] = old_num_scheduled_tokens
```

### 3.4 `_record_echo_overcount`

在 `propose_draft_token_ids` 内 `_copy_valid_sampled_token_count` 之后调用：

```python
over = old_sched - valid_count  # echo_prev_scheduled - verify 实际接受数
if over > 0:
    self.echo_overcount_pending[req_id] = over
self.echo_prev_scheduled = {}
```

下一步 `_update_states` 从 scheduler 的 `num_computed_tokens` 减去 `over`，修正 async scheduling 按 **trim 前** 调度量（如 8）乐观推进、但 worker 实际只处理 **trim 后** 量（如 3，接受 1）导致的漂移（如 2673+8=2681）。

### 3.5 `_prepare_input_ids`（override）

调用 `super()._prepare_input_ids(...)` 后，ECHO 路径额外：

1. 将 trim 后的 `spec_tokens` 写入 `input_ids_cpu[spec_start:flat_end+1]`
2. 从 `prev_sampled_token_ids` 写入 sample 位置
3. **`echo_keep=0` 时**（无 spec token、仅 1 个 sample token）同样写入 sample 位
4. 若 `prev_sampled <= 0`，回退到 `token_ids_cpu` 最后一个有效 output token
5. 按需 `copy_to_gpu`

**动机**：async spec-decode 的 base scatter 无法正确处理 ECHO trim 后的变长 draft；`echo_keep=0` 时 `scheduled_spec_decode_tokens` 为空会导致 early return 漏写 sample。

### 3.6 `_get_draft_token_ids_cpu`（override）

ECHO 启用时过滤 `-1`，返回 `(filtered_rows, req_ids)`。

### 3.7 `propose_draft_token_ids` 闭包内

propose 完成后：

```python
mask = (self._draft_token_ids != -1)
kept_counts = mask.int().sum(dim=1).cpu().tolist()
filtered_drafts, draft_req_ids = self._get_draft_token_ids_cpu()
req_ids_for_echo = draft_req_ids if draft_req_ids else self.input_batch.req_ids
self.echo_cu_draft_tokens = dict(zip(req_ids_for_echo, kept_counts))
self.echo_draft_tokens = dict(zip(req_ids_for_echo, filtered_drafts))
```

## 4. 已移除 / 未采用的代码

### `_sync_echo_uniform_decode_len`

曾将 `uniform_decode_query_len` 设为 1，与 `FULL_DECODE_ONLY` CUDAGraph padding 冲突。**已删除**。

### `update_num_computed_tokens` 全量 participating 方案（已回滚）

曾将 `participating = (prev_positions >= 0)` 用于所有 continuing 请求，导致 `prev_drafts==0` 时 `prev_gpu + valid_count` 与 CPU 严重偏离，M-RoPE drift 产生 **负 position**。**已回滚**，仍用 `(prev_positions >= 0) & (prev_drafts > 0)`；prefill 后首步 decode 的计数漂移改由 **过计数修正** 处理。
