# 03 - 排查思路与关键发现

## 排查时间线

### Phase 1: 怀疑slot_mapping计算错误 ❌

**现象**: ECHO的slot(31345)与基线(19057)不同

**分析**:
- 31345 = 244×128 + 113 (position 2673)
- 19057 = 148×128 + 113 (position 2673)
- 同position、不同物理块 → **两次run的block池分配顺序不同**

**结论**: slot公式计算正确，不是根因。

---

### Phase 2: 怀疑input_ids scatter错误 ❌

**现象**: 怀疑剪枝后的draft有`-1`空洞，scatter时读错

**分析**:
- ECHO的cum_log_probs单调不增 → 剪枝保留前缀，不会有中间洞
- 用户确认`input_ids`里没有`-1`
- 基线同样`input_ids=None`也能正常跑

**结论**: input_ids不是根因。

---

### Phase 3: 确认NaN出现在step3 forward之后 ⚠️

**关键发现**:
- NaN在step3的`_model_forward`返回后就存在
- slot/position/input都正确 → 问题在**attention/KV路径**

**当前排查重点**:

#### 检查1: seq_lens同步 (最可疑)

```python
# _prepare_inputs中
seq_lens_gpu = num_computed_tokens + num_scheduled_tokens_gpu
optimistic_seq_lens_cpu = num_computed_tokens_cpu + num_scheduled_tokens  # 乐观估计

# async模式下，_build_attention_metadata使用:
max_seq_len = optimistic_seq_lens_cpu.numpy()[:num_reqs].max().item()
```

**风险点**:
- ECHO step3只schedule 7 token(4+3)，但乐观估计可能按trim前的8 token算
- seq_lens_cpu > seq_lens_gpu → attention mask范围错误 → 读未初始化KV → NaN

#### 检查2: KV cache写入完整性

**问题**: step2的mixed batch `[req0 decode, req1 prefill]` 中，req1的2673 token是否都写入了KV？

**验证方法**:
- 对比step2末和step3初的`block_table_row`是否完全一致
- 检查`seq_lens`与`num_computed`是否匹配

#### 检查3: attention state一致性

```python
# _build_attn_state逻辑
if np.all(num_scheduled_tokens == 1):
    attn_state = DecodeOnly / SpecDecoding
elif np.all(num_valid_tokens == 1):
    attn_state = SpecDecoding  # 或 ChunkedPrefill
```

ECHO step3: `num_scheduled=[4,3]` → 不是纯decode → 走`ChunkedPrefill`或`SpecDecoding`路径

需要确认attn_state与实际的KV布局是否匹配。
