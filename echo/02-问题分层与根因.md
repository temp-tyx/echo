# 02 — 问题分层与根因

[← 返回目录](./README.md)

## 统一根因（一句话）

**所有症状都源于同一件事：ECHO 实际 verify 的宽度（最多 = 投机步数,约 6 token）与 vLLM 各组件按 `num_speculative_tokens` 静态分配的宽度不一致。**

排查过程中曾以为是三个独立 layer、需要分层 patch；最终发现它们是**同一根因的不同投影**，正确解法是让 `num_speculative_tokens` 本身 = 最大投机宽度，一次性把所有派生宽度对齐。

```text
症状                      表层定位              真实根因
─────────────────────────────────────────────────────────────
multi-concurrency NaN     KV / slot_mapping     verify 宽度 > 已写 KV → 读到未写脏页
单并发重复 token          schedule 账本          sync 缺失 + buffer 太窄
输出质量差 / 语句不通       attention 路径         decode_threshold 太小 → 6-token 被当 prefill
eager MTE DDR 越界         attention kernel       buffer/kernel 按 3 分配,实算 6 → 越界
```

---

## Layer 1：multi-concurrency KV 脏页 / slot_mapping 漂移

### 现象
- multi-concurrency 乱码、NaN
- 同 prompt/batch，ECHO 与基线 **slot_mapping 不同**（同 logical block，不同 physical block）

### 根因
两个因素叠加：
1. **尺寸不足**：按 `num_speculative_tokens=3` 分配 KV / buffer，ECHO 实算 6 token。
2. **scheduled 宽度 ≠ 已写 KV 宽度**：scheduler 按 placeholder 预留了 N 个 slot，但 ECHO prune 后实际只写了 M(<N) 个，attention 仍按 seq_len 读到**未写入的脏 slot（NaN）**。

### 最终修复
- `num_speculative_tokens = max`：所有进程、所有组件**启动时**按同一上限分配，消除尺寸不足。
- **sync 真实 draft 数**回 scheduler：scheduled 宽度 == verify 宽度 == 已写 KV 宽度，不再读未写 slot。
- `_copy_draft_token_ids_to_cpu`：按当步真实列数拷贝、尾列补 `-1`，兼容多并发下动态 draft 宽度（修掉 `tensor(7) vs (5)` copy 报错）。

---

## Layer 2：单并发重复输出

### 现象
- 单并发输出重复 token；`num_computed` 异常回退；`num_accepted=0`

### 根因
- `draft_token_ids_cpu` 只有 3 列 → sync 到 engine 的 `spec_token_ids` 被截断。
- 旧 **execute_model trim** 在 worker 本地把 `num_scheduled` 扩到 6，但 engine `update_from_output` 仍按**未 trim** 的原始 schedule 算 rejection → 双账本错乱。

### 最终修复
| 改动 | 作用 |
|------|------|
| `num_speculative_tokens = max` → buffer 自然够宽 | sync 不被截断 |
| `take_draft_token_ids` 过滤 `-1` | engine 只收有效 draft |
| **删除 execute_model trim** | 不再 worker/engine 双账本 |
| `patch_echo_async.py` | async 下 schedule 前 sync draft |
| `_compact_echo_draft_rows` | prune 后左对齐，scatter 前缀正确 |

---

## Layer 3：输出质量差 / eager 崩溃

### 现象
- 重复消失后语义混乱（如 `"饮料瓶代码与人物微光——"`），不像 NaN，更像**错误 draft 被 accept**。
- `--enforce-eager` 下直接 **MTE error: DDR address out of range**（graph 模式把它掩盖了）。

### 根因
`split_decodes_and_prefills` 判定：

```python
if max_query_len <= decode_threshold:
    return num_reqs, 0, num_tokens, 0   # 全部当 decode
# 否则当 prefill
```

当 `num_speculative_tokens=3` 时 `decode_threshold=4`，而 6-token spec verify 的 `max_query_len=6`：`6 > 4` → **6-token decode 被误判 prefill** → kernel/metadata 错误（eager 下直接越界，graph 下静默错算）。

### 最终修复
不再 runtime patch `decode_threshold`。直接让 `num_speculative_tokens=max(=7)`，于是 vLLM 原生派生 `decode_threshold = 1+7 = 8`，`6 ≤ 8` 自动走 decode。

---

## 为什么"保持 config=3 + runtime patch"被废弃

曾经的思路是：保持 `num_speculative_tokens=3`（怕动 KV），再 runtime 逐个把派生宽度提到 8：
`uniform_decode_query_len`、`decode_threshold`(runner + 每个 attention builder)、`decode_token_per_req`、`cudagraph_dispatcher`、`compiler_interface` 捕获范围……

问题：

1. **whack-a-mole**：派生宽度散落在多处，补一个冒一个；`--enforce-eager` 下仍 MTE 越界,说明还有没补到的 buffer/kernel。
2. **时序坑**：很多派生值在 drafter 创建**之前**就算好了，runtime 再改是补丁叠补丁。
3. **进程不一致**：worker 改的值，engine 进程的 scheduler 看不到。

最终结论：**与其在 runtime 追着对齐，不如让 `num_speculative_tokens` 一开始就 = 最大宽度**，所有派生值由原生逻辑统一算对。代码更简单、eager/graph 都正确。

---

## 已排除的方向

| 方向 | 原因 |
|------|------|
| topk 中间挖洞（bs=1） | cum log-prob 单调时 top-k 必为前缀，不会挖洞 |
| worker forward input_ids 错位 | 用户 log 确认 shape/position 对齐 |
| execute trim 长期保留 | 只改 worker、不改 engine 账本，双账本必错 |
| config=3 + runtime patch 宽度 | whack-a-mole + eager 越界，见上 |
