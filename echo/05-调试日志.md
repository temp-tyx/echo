# 05 - 调试日志使用说明

## 开启调试

```bash
export VLLM_ECHO_DEBUG=1
python your_script.py 2>&1 | tee echo.log
```

## 日志过滤

```bash
# 只看关键节点
grep -E "ECHO \[(slot_mapping|prepare|attn|forward)\]" echo.log

# 只看req1
grep "1-" echo.log | grep ECHO

# 看NaN情况
grep "has_nan" echo.log
```

## 日志字段详解

### [slot_mapping]

| 字段 | 示例 | 说明 |
|------|------|------|
| req_ids | `['0-abc', '1-def']` | 当前batch的请求ID |
| num_scheduled | `[4, 3]` | 每个request的token数 |
| token_layout | 见下 | 每个request的position/slot详情 |

**token_layout字段**:
- `pos_first_last`: [首position, 尾position]
- `slot_first_last`: [首slot, 尾slot]
- `pos_drift_nonzero`: position漂移的非0个数(应为0)

### [prepare]

| 字段 | 示例 | 说明 |
|------|------|------|
| attn_state | `SpecDecoding` | attention计算模式 |
| seq_lens_gpu | `[2680, 2676]` | GPU上实际的seq_len |
| seq_lens_cpu | `[2680, 2676]` | CPU乐观估计的seq_len |
| seq_drift | `[0, 0]` | 差异(应为0) |

**关键检查**: `seq_drift`必须全0，否则attention mask会错。

### [attn]

| 字段 | 示例 | 说明 |
|------|------|------|
| max_seq_len | `2680` | attention使用的最大seq_len |
| num_actual_tokens | `7` | 实际token数 |

### [forward]

| 字段 | 示例 | 说明 |
|------|------|------|
| has_nan | `true/false` | 是否有NaN |
| nan_count | `768` | NaN元素个数 |
| token_range | `[4, 7]` | 在batch中的位置 |

## 对比基线的方法

```bash
# 基线run (不开ECHO)
export VLLM_ECHO_ENABLED=0
python your_script.py 2>&1 | tee baseline.log

# ECHO run
export VLLM_ECHO_ENABLED=1
python your_script.py 2>&1 | tee echo.log

# 对比step3的关键字段
grep "ECHO \[prepare\]" baseline.log | tail -1 > b_prepare.txt
grep "ECHO \[prepare\]" echo.log | tail -1 > e_prepare.txt
diff b_prepare.txt e_prepare.txt
```

## 常见问题模式

### 模式1: seq_drift非0
```
seq_lens_gpu=[2680, 2676]
seq_lens_cpu=[2681, 2676]  # <-- 多了1
seq_drift=[-1, 0]
```
**解读**: scheduler乐观估计与GPU实际状态不一致，可能是trim后未同步。

### 模式2: req1的has_nan=true
```
per_req=[
  {'req_id': '0-xxx', 'has_nan': false},
  {'req_id': '1-yyy', 'has_nan': true, 'nan_count': 768}
]
```
**解读**: req1的hidden_states全NaN，需往前追溯step2的KV写入。

### 模式3: slot不连续
```
'slot_first_last': [31345, 31347]  # <-- 差2，但num_tokens=3
```
**解读**: 正常应该是[31345, 31347]对于3个token。若差值!=num_tokens-1，则slot_mapping有误。
