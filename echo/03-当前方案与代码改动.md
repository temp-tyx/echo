# 03 — 当前方案与代码改动

[← 返回目录](./README.md)

## 设计原则

1. **init 不改 `vllm_config.speculative_config.num_speculative_tokens`**（保持 MTP3=3，KV/lookahead 与基线一致）
2. **draft 宽度**在 proposer 内扩展（buffer、propose 步数）
3. **engine schedule 宽度**由 propose 后 **sync pruned draft** 决定，不在 execute 里 trim
4. **runtime decode 宽度**（attention/graph）单独扩到 `1 + VLLM_ECHO_MAX_SPEC_NUM`，与 config 解耦

## 架构图

```text
                    ┌─────────────────────────────────────┐
                    │  vllm_config.num_speculative_tokens=3 │  ← 不动（KV/lookahead）
                    └─────────────────────────────────────┘
                                      │
        ┌─────────────────────────────┼─────────────────────────────┐
        ▼                             ▼                             ▼
   draft propose              sync → spec_token_ids          runtime decode 宽度
   (7 列, prune→5)           (过滤 -1, len=5)              (decode_threshold=8)
        │                             │                             │
        └─────────────────────────────┴─────────────────────────────┘
                                      ▼
                         schedule: 1+5=6, verify 走 decode 路径
```

## 关键改动清单

### `vllm_ascend/spec_decode/eagle_proposer.py`

- init **注释掉** `vllm_config.speculative_config.num_speculative_tokens = 7`
- drafter 内部 `num_speculative_tokens = max(ECHO_MAX, config)`
- `_apply_echo_pruning` + `_compact_echo_draft_rows`（左对齐有效 draft）
- `_echo_per_req_draft_counts`

### `vllm_ascend/worker/model_runner_v1.py`

| 功能 | 说明 |
|------|------|
| init 扩 `draft_token_ids_cpu` | `[max_reqs, VLLM_ECHO_MAX_SPEC_NUM]` |
| `_copy_draft_token_ids_to_cpu` | ECHO+async 始终拷贝全宽 draft |
| `take_draft_token_ids` | 过滤 `-1` |
| **删除** execute_model trim | 不再改 `scheduler_output` |
| `propose_draft_token_ids` | 不写 `speculative_config`；propose 后恢复 drafter 宽度 |
| `_apply_echo_runtime_decode_width` | `uniform_decode_query_len` / `decode_threshold` → 8 |
| `_patch_attn_builders_echo_decode_width` | attention builder 同步 decode_threshold |
| target_forward diag log | 可选，验证用 |

### `vllm_ascend/patch/platform/patch_echo_async.py`

- `EngineCore.step_with_batch_queue`：schedule **前** sync draft
- `EngineCore.post_step`：ECHO+async 在 model 执行后 sync
- 非 async 仍走 vLLM 原有 `post_step` → `take_draft_token_ids`

### `vllm_ascend/compilation/compiler_interface.py`

- static kernel 捕获范围按 `max(MTP3, ECHO_MAX)` 计算

### 不再推荐

- execute_model 里按 `echo_cu` trim `scheduler_output`
- init 全局改 `vllm_config.num_speculative_tokens`
- propose 里 `self.speculative_config.num_speculative_tokens = draft_step`（污染共享 config）

## propose 路径注意点

```python
draft_step = min(max(2 * k_max / batch_size, 1), draft_max)
self.drafter.num_speculative_tokens = draft_step   # 仅 propose 期间
# 不要写 vllm_config / speculative_config
# propose 结束后恢复 drafter 宽度，self.num_spec_tokens = draft_tensor.shape[1]（scatter stride）
```

---

## async vs sync draft sync

| 模式 | sync 时机 |
|------|-----------|
| sync | `EngineCore.post_step` → `take_draft_token_ids` → `update_draft_token_ids` |
| async | `patch_echo_async`：schedule 前 + post_step |

async 下若不同步，engine 仍用 placeholder 宽度 schedule，placeholder 账本会漂移。

## 性能说明

| 项 | 开销 |
|----|------|
| CPU buffer 7 列 vs 3 列 | 每 req 多几个 int，可忽略 |
| sync + 过滤 `-1` | 原链路已有，略增 list 长度 |
| 去 trim | 略减 CPU 操作 |
| verify 5 draft vs MTP3 | ECHO 算法本身开销，与 sync 无关 |

相对 init=7：算力路径类似（每步 ~6 token verify），**KV lookahead 仍按 3**，multi-concurrency 更省。
